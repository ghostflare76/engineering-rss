GitHub, MCP를 리눅스 재단에 기부하며 AI 에이전트 개발 표준 제시

LLM SDK 사용량 178% 증가, 37K 스타 달성, GitHub Copilot 100만+ PR 자동 생성

요즘 AI 개발 속도가 정말 엄청나죠. 지난 한 해 동안에만 110만 개가 넘는 공개 GitHub 저장소에서 LLM SDK를 가져다 쓰고(전년 대비 +178%), 약 70만 개의 새로운 AI 저장소가 생겨났다고 합니다. vllm, ollama, continue 같은 에이전트 도구들은 이제 개발자 스택의 필수 요소가 되고 있고요.

그런데 이렇게 폭발적으로 성장하는 AI 생태계에서 모델을 외부 도구와 시스템에 안전하고 일관적이며, 또 플랫폼에 구애받지 않고 연결하는 문제가 늘 발목을 잡곤 했습니다. 마치 모든 모델 클라이언트가 모든 시스템과 개별적으로 통합해야 하는 'n×m 통합 문제'에 부딪히는 느낌이었죠.

바로 이런 격차를 빠르게 메워준 것이 Model Context Protocol (MCP)입니다. Anthropic에서 오픈소스 아이디어로 시작된 MCP는 GitHub와 Microsoft 같은 기업들도 합류하며 업계에서 가장 빠르게 성장하는 표준 중 하나가 되었는데요. 이제 MCP가 리눅스 재단의 관리 아래 Agentic AI Foundation에 기부되며 새로운 공유 리더십 시대를 맞았습니다. 이 글에서는 MCP가 어떻게 여기까지 왔고, 이번 리눅스 재단으로의 이관이 AI 개발의 미래에 어떤 의미를 가지는지 자세히 살펴볼 겁니다.

  • AI 개발의 폭발적 성장과 통합 문제 - 178% 증가한 LLM SDK 사용량과 에이전트 도구의 부상
  • MCP 등장 이전의 혼돈 - 파편화된 API, 예측 불가능한 통합, n×m 문제의 고통
  • MCP의 탄생과 핵심 가치 - Anthropic의 내부 패턴에서 시작된 오픈소스 표준 프로토콜
  • 빠른 확산의 비결 - OAuth, Registry, 장기 실행 태스크 지원을 통한 실전 워크플로우 개선
  • 리눅스 재단 합류의 의미 - 프로토콜이 모두를 위한 '진정한 인프라'가 되는 과정
  • 지금 바로 MCP가 개발자에게 제공하는 6가지 실질적인 이점 - 모델 연결의 미래 청사진

💥 AI 개발의 폭발적 성장과 통합의 숙제

지난 한 해 동안 AI 개발은 그야말로 폭발했습니다. 2025년 옥토버스 보고서에 따르면, 110만 개가 넘는 공개 GitHub 저장소에서 LLM SDK를 가져다 쓰고 있으며(전년 대비 178% 증가), 거의 70만 개의 새로운 AI 저장소가 만들어졌다고 합니다. vllm, ollama, continue, aider, ragflow, cline 같은 에이전트 도구들은 이제 현대 개발자 스택의 핵심 부분으로 빠르게 자리 잡고 있습니다.

이처럼 생태계가 확장되면서, 우리는 모델을 외부 도구 및 시스템에 안전하고 일관성 있게, 그리고 플랫폼을 넘나들며 연결해야 할 필요성을 절실히 느끼게 되었습니다. Model Context Protocol (MCP)은 바로 이 간극을 빠르게 채워줬죠.

Anthropic의 오픈소스 아이디어로 시작된 MCP는 처음부터 개방적으로 설계되어 커뮤니티가 함께 확장하고 채택하며 형성할 수 있었기에 빠르게 성장했습니다. GitHub와 Microsoft 같은 회사들도 이 표준을 함께 구축하는 데 참여하면서, MCP는 업계에서 가장 빠르게 성장하는 표준 중 하나가 될 수 있었어요.

그리고 이제, Anthropic은 MCP를 Agentic AI Foundation에 기부했고, 이 재단은 리눅스 재단의 관리 아래 놓이게 되었습니다. 이는 MCP가 장기적인 툴링, 프로덕션 에이전트, 그리고 엔터프라이즈 시스템을 위한 견고한 기반을 제공하며 새로운 공유 리더십 시대로 접어들었음을 의미합니다.

💔 MCP 이전: 산산조각 난 API와 예측 불가능한 통합

초기의 LLM들은 고립된 시스템이었습니다. 프롬프트를 보내면 응답을 받는 식이었죠. 우리는 RAG(Retrieval-Augmented Generation) 같은 패턴을 사용해 LLM에 더 많은 문맥 데이터를 제공했지만, 한계가 명확했습니다. OpenAI의 함수 호출(Function Calling) 도입은 외부 함수를 호출할 수 있게 하면서 큰 변화를 가져왔고, GitHub Copilot 팀도 이를 기반으로 초기 시스템을 구축했습니다.

하지만 2023년 초까지 개발자들은 LLM을 외부 시스템에 연결하기 위해 호환되지 않는 API들의 '조각 패치워크'에 의존하고 있었습니다. 맞춤형 확장 기능, IDE 플러그인, 플랫폼별 에이전트 프레임워크 등이 난무했죠. 모든 제공업체가 각자의 통합 방식을 가지고 있었고, 어느 하나도 똑같이 작동하는 것이 없었습니다.

문제는 툴링이 아니라 아키텍처였다

LLM을 실시간 웹, 데이터베이스, 티켓팅 시스템, 검색 인덱스, CI 파이프라인에 연결하려면 모델 업데이트 시 자주 깨지는 맞춤형 코드가 필요했습니다. 개발자들은 플랫폼마다 깊이 있는 통합 코드를 일일이 작성해야 했죠.

Anthropic의 수석 엔지니어이자 MCP의 원 설계자 중 한 명인 David Soria Parra는 당시 상황을 'n×m 통합 문제'로 설명했습니다. 너무 많은 클라이언트와 너무 많은 시스템이 있었고, 이들을 연결할 공유 프로토콜이 없었다는 것이죠. 예를 들어, 5개의 AI 클라이언트가 10개의 내부 시스템과 통신하려면 50개의 맞춤형 통합이 필요했고, 각각 다른 의미론, 인증 흐름, 오류 모드를 가졌습니다. MCP는 단일하고 벤더 중립적인 프로토콜을 정의하여 이 문제를 해결합니다.

표준의 부재는 비효율적일 뿐 아니라 실제 채택을 늦췄습니다. 금융, 헬스케어, 보안과 같은 규제 산업에서는 모델이 시스템과 통신하는 데 안전하고 감사 가능하며 플랫폼 간 호환되는 방식이 필요했지만, 그들이 얻은 것은 불분명한 신뢰 경계를 가진 독점적인 플러그인 생태계였습니다.

🛠️ MCP의 등장: 개발자 워크플로우를 위한 표준 프로토콜

Anthropic, GitHub, Microsoft 등 업계 전반의 엔지니어들은 모델을 컨텍스트 및 도구에 안정적으로 연결하는 동일한 문제에 직면했습니다. Anthropic 내부 팀은 데이터 요청, 도구 호출, 장기 실행 작업 처리와 관련하여 자체 프로토타입이 유사한 패턴으로 수렴하는 것을 발견했죠.

Soria Parra는 MCP의 시작을 "Anthropic 엔지니어들이 끊임없이 재발명하던 패턴을 표준화하는 방법"으로 설명했습니다. MCP는 이러한 패턴을 모델과 시스템이 서로 통신하고, 컨텍스트를 요청하며, 도구를 실행하는 방식인 '커뮤니케이션'에 중점을 둔 프로토콜로 증류했습니다.

Anthropic의 Jerome Swanwick은 초기 내부 해커톤에서 "모든 출품작이 MCP를 기반으로 만들어져 내부적으로 큰 반향을 일으켰다"고 회상했습니다. 이러한 초기 개발자들의 관심이 씨앗이 되었고, Anthropic이 MCP를 공개하고 고품질 참조 서버를 함께 출시하자마자 커뮤니티는 그 가치를 즉시 알아봤습니다. MCP는 클라이언트, 런타임 또는 벤더에 관계없이 모델이 외부 시스템과 통신할 수 있는 공유 방식을 제공했으니까요.

그림 1. MCP가 GitHub 블로그에서 리눅스 재단으로 향하기까지의 여정 (유튜브 영상 썸네일)

✨ MCP가 가진 3가지 핵심 강점: 왜 이렇게 빠르게 확산됐을까?

MCP가 출시되자마자 채택 속도는 전례 없이 빨랐습니다. AI 기반 도구와 에이전트를 구축하는 개발자들은 이미 MCP가 해결하려는 고통을 겪고 있었기 때문이죠. Microsoft의 수석 엔지니어이자 MCP 운영 위원회 멤버인 Den Delimarsky는 "그것은 즉시 이해됐다. 그들이 해결하려는 문제와 왜 이것이 필요한지 알았다"고 말했습니다.

몇 주 만에 Anthropic, Microsoft, GitHub, OpenAI 및 독립 개발자들이 프로토콜을 확장하고 강화하기 시작했습니다. 다음 9개월 동안 커뮤니티는 다음을 추가했습니다.

안전한 원격 서버를 위한 OAuth 흐름

일관된 모델 동작을 위한 샘플링 의미론

정교해진 도구 스키마 및 서버 검색 패턴

장기 실행 태스크 지원 강화

특히 장기 실행 태스크 API는 중요한 기능입니다. 빌드, 인덱싱 작업, 배포 등 몇 분이 걸리는 작업들을 폴링(polling) 해킹이나 맞춤형 콜백 채널 없이 예측 가능하게 추적할 수 있게 해주기 때문입니다. 이는 오늘날 우리가 보는 장기 실행 AI 에이전트 워크플로우에 필수적이었습니다.

Delimarsky의 OAuth 작업도 전환점이 되었습니다. 이전에는 대부분의 MCP 서버가 로컬에서 실행되어 엔터프라이즈 환경에서의 사용이 제한적이었고 설치 마찰을 유발했습니다. OAuth는 원격 MCP 서버를 가능하게 하여 대규모의 안전하고 규정을 준수하는 통합을 가능하게 했습니다. 이 변화는 MCP를 다중 머신 오케스트레이션, 공유 엔터프라이즈 서비스 및 비로컬 인프라에 사용할 수 있도록 만들었습니다.

또한 MCP Registry는 Anthropic, GitHub 등이 기여하고 툴링을 지원하여 개발자에게 발견 가능성 계층을 제공하고 기업에게는 거버넌스 제어권을 주었습니다. GitHub에서 MCP 서버 및 Registry를 이끄는 Toby Padilla는 이를 "개발자들이 고품질 서버를 찾고, 기업이 사용자의 채택을 제어할 수 있는 방법"이라고 설명했습니다.

그러나 MCP의 궤적을 이끈 것은 단일 회사가 아니었습니다. 제가 커뮤니티와의 모든 대화에서 가장 인상 깊었던 것은 '공유된 리더십'에 대한 느낌이었습니다. Nick Cooper(OpenAI 엔지니어이자 MCP 운영 위원회 멤버)는 "저는 Anthropic과 만나는 것이 아니라 David와 만나는 것이고, Google과 만나는 것이 아니라 Che와 만나는 것"이라고 명확하게 표현했습니다. 작업은 결코 기업의 경계에 관한 것이 아니라 프로토콜 자체에 관한 것이었습니다. 이러한 협력적 문화는 웹 초기의 모습을 연상시키며 오픈소스의 가장 좋은 면모를 보여주었고, 이것이 MCP가 그토록 빠르게 확산된 핵심 이유라고 생각합니다.

🌐 리눅스 재단 합류: 프로토콜이 인프라가 되는 순간

MCP 채택이 가속화되면서 중립적인 거버넌스의 필요성이 대두되었습니다. 개방성은 초기 채택을 이끌었지만, 여러 LLM 제공업체, 도구 빌더, 엔터프라이즈 팀이 프로토콜에 의존하게 되면서 이는 공유된 리더십을 요구하게 되었습니다.

Anthropic과 MCP 운영 위원회는 거버넌스를 리눅스 재단으로 전환함으로써, MCP가 진정한 산업 표준으로서의 성숙도 임계값에 도달했음을 알리고 있습니다. 개방적이고 벤더 중립적인 거버넌스는 모든 이해관계자에게 다음과 같은 이점을 제공합니다.

1. 장기적인 안정성: 프로토콜의 강점은 수명에 있습니다. 리눅스 재단의 지원은 MCP를 깊이 통합하는 팀의 위험을 줄여줍니다.

2. 공평한 참여: 클라우드 제공업체든, 스타트업이든, 개별 관리자든, 리눅스 재단의 거버넌스 프로세스는 공평한 기여 권한과 투명한 진화를 지원합니다.

3. 호환성 보장: 더 많은 클라이언트, 서버, 에이전트 프레임워크가 MCP에 의존함에 따라 호환성은 프로토콜 자체만큼 중요해집니다.

4. 오픈 표준의 안전성: AI가 규제 대상 워크로드의 일부가 되는 시대에, 중립적인 거버넌스는 MCP를 기업에게 더 안전한 선택으로 만듭니다.

MCP는 이제 Kubernetes, SPDX, GraphQL, CNCF 스택과 같은 기술처럼, 오픈 소스로 유지 관리되는 중요한 인프라의 길을 걷게 되었습니다. 이러한 움직임은 Agentic AI Foundation이 여러 모델 제공업체, 플랫폼 팀, 엔터프라이즈 도구 빌더 및 독립 개발자들을 공유되고 중립적인 프로세스 아래 통합하려는 의도와 일치합니다.

🚀 지금 바로 MCP가 개발자에게 제공하는 6가지 가치

개발자들은 종종 "MCP를 채택하면 실제로 무엇을 얻을 수 있나요?"라고 묻곤 합니다. 제가 생각하는 MCP의 구체적인 가치는 다음과 같습니다.

1. 하나의 서버, 여러 클라이언트: 한 번 도구를 노출하면 여러 AI 클라이언트, 에이전트, 셸, IDE에서 사용할 수 있습니다. 더 이상 모델 제공업체마다 맞춤형 함수 호출 어댑터를 만들 필요가 없습니다.

2. 예측 가능하고 테스트 가능한 도구 호출: MCP의 스키마는 도구 상호 작용을 디버그 가능하고 신뢰할 수 있게 만들어, 프롬프트 엔지니어링보다 API 계약에 가깝게 만듭니다.

3. 에이전트 네이티브 워크로드를 위한 프로토콜: 옥토버스 보고서에서 보듯이, 에이전트 워크플로우는 주류 엔지니어링으로 진입하고 있습니다. 에이전트는 도구를 호출하고 컨텍스트를 가져오는 구조화된 방식이 필요하며, MCP가 바로 그것을 제공합니다.

4. 안전한 원격 실행: OAuth 및 원격 서버 지원 덕분에 MCP는 기업, 규제 대상 워크로드, 다중 머신 오케스트레이션, 공유 내부 도구 등에서 작동합니다.

5. 성장하는 서버 생태계: 커뮤니티 및 벤더가 유지 관리하는 MCP 서버(매주 추가됨)를 통해 개발자들은 이슈 트래커, 코드 검색, 관찰 시스템, 내부 API, 클라우드 서비스, 개인 생산성 도구 등에 연결할 수 있습니다.

6. 개발자가 이미 소프트웨어를 구축하는 방식과 일치: MCP는 스키마 기반 인터페이스(JSON Schema 기반), 재현 가능한 워크플로우, 컨테이너화된 인프라, CI/CD 환경 등 개발자 습관과 일치합니다. 대부분의 개발자는 예측 가능한 시스템을 원하며, MCP는 이러한 기대를 충족시킵니다.

Soria Parra는 MCP가 단순히 LLM이 도구를 호출하는 데 그치지 않고, 개발자가 모델을 사용하여 자신의 복잡한 시스템을 이해할 수 있도록 워크플로우를 '역전'시킬 수도 있다고 강조했습니다. MCP는 '마법 같은' 모델 동작보다는 예측 가능하고 계약 기반의 상호 작용을 선호하며, 개발자들이 이미 API 설계, 분산 시스템, 표준 진화에서 알고 있는 패턴을 의도적으로 반영합니다.

실전 적용 인사이트

MCP는 단순한 프로토콜이 아니라 AI와 외부 시스템을 연결하는 새로운 표준입니다. Anthropic에서 시작되어 GitHub, Microsoft 등 다양한 커뮤니티가 함께 발전시켰고, 이제 리눅스 재단의 중립적 거버넌스 아래 장기적인 안정성과 신뢰성을 확보했습니다. AI 에이전트 개발의 핵심 인프라가 될 MCP의 미래를 함께하세요.
